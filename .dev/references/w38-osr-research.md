# W38: OSR + SIMD State Transfer Research

Investigation for enabling JIT of C-compiled wasm functions with reentry guards.

## Problem Statement

C-compiled functions (`wasm32-wasi-clang -O2 -msimd128`) have reentry guards
(`__cxa_atexit` init pattern) that prevent back-edge JIT. Hot loops run entirely
in the register IR interpreter, causing 13-131x slowdown vs wasmtime.

OSR (On-Stack Replacement) can bypass the guard, but SIMD v128 state doesn't
transfer correctly from interpreter to JIT for complex C functions.

## How Other Runtimes Handle This

### wasmtime: No tier-up, AOT only

- **No OSR, no interpreter**: Compiles everything with Cranelift (optimizing) or
  Winch (baseline) directly to native code. The problem doesn't exist.
- **SIMD**: v128 is just another Cranelift `Value` (type `I8x16`). No separate
  storage — handled entirely by register allocator. Bitcasts between lane
  interpretations are coalesced away.
- **Reentry guards**: Not relevant — all init runs at module instantiation time.
- **Key insight**: Single-tier AOT avoids all state transfer issues.

### V8 (Chrome): No wasm OSR, function-level tier-up only

- **Two tiers**: Liftoff (baseline) → TurboFan (optimizing).
- **NO OSR for wasm**: "We don't do on-stack-replacement. If TurboFan code
  becomes available after the function was called, the function call will
  complete its execution with Liftoff code."
  ([v8.dev/docs/wasm-compilation-pipeline](https://v8.dev/docs/wasm-compilation-pipeline))
- **Tier-up**: Triggered by call count threshold. Background thread compiles
  with TurboFan. Next call uses optimized code. Current call continues in Liftoff.
- **Deopt (TurboFan → Liftoff)**: V8 DOES support wasm deoptimization since 2025.
  The deoptimizer reconstructs Liftoff frame state from TurboFan's optimized
  state using `DeoptimizationData` metadata generated by Liftoff during compilation.
  "Due to the SIMD extension for Wasm this also includes all 128-bit vector
  registers used by TurboFan."
  ([v8.dev/blog/wasm-speculative-optimizations](https://v8.dev/blog/wasm-speculative-optimizations))
- **Key insight**: Deopt metadata approach — the baseline compiler generates
  a frame layout descriptor for each PC, and the deoptimizer uses it to map
  optimized state → baseline state. This is essentially the REVERSE of our
  problem (they go TurboFan→Liftoff, we need interpreter→JIT).

### SpiderMonkey (Firefox): Function-level tier-up via jump table

- **Two tiers**: Wasm Baseline → Ion.
- **NO OSR**: "SpiderMonkey can only tier up at function calls — it doesn't
  do OSR." All calls go through a jump table indirection to allow per-function
  tier-up. ([wingolog.org/archives/2020/03/25](http://wingolog.org/archives/2020/03/25/firefoxs-low-latency-webassembly-compiler))
- **Key insight**: Avoids OSR entirely by only tier-up at function boundaries.

### WAMR: Multi-tier JIT, function-level switch

- **Three modes**: Interpreter → Fast JIT → LLVM JIT.
- Tier-up happens at function boundaries (no OSR documented).
- No public documentation on state transfer mechanisms.

## Approaches for zwasm

Given the research, here are candidates ordered by feasibility:

### Approach A: Lazy AOT (Recommended — Phase 18.2)

**Idea**: Compile all functions to JIT at module load time (or lazily on first
call), bypassing the interpreter entirely for C-compiled modules.

**Pros**:
- Eliminates the tier-up problem entirely (matches wasmtime's approach)
- Reentry guard is irrelevant — JIT code handles it natively (guard_branch_pc)
- Already partially designed in Phase 18.2 (Lazy Compilation)

**Cons**:
- Startup overhead for large modules (hundreds of functions)
- Functions that exceed MAX_JIT_IR_INSTRS still need interpreter

**Implementation**:
1. At module instantiation, compile all functions with JIT (background thread)
2. On first call, if JIT code is ready → use it; otherwise → interpret
3. JIT entry from call (not from back-edge) avoids OSR entirely
4. Reentry guard handled by guard_branch_pc (already implemented)
5. Side-effect-free: function runs from pc=0 with fresh register state

**Estimated effort**: Medium — core mechanism exists, need call-site tier check.

### Approach B: Frame State Metadata (V8-style deopt map)

**Idea**: The JIT compiler generates a "frame layout map" for each back-edge
target PC. At OSR entry, the map describes where each value (scalar, f32, v128)
should be placed. The OSR prologue uses this map to correctly transfer state.

**Pros**:
- Correct by construction — type-aware state transfer
- Handles interleaved scalar/SIMD vregs correctly

**Cons**:
- Significant complexity: need to track value types through JIT compilation
- Memory overhead for storing maps per function per back-edge
- Only V8 has done this, and only for deopt (reverse direction)

**Implementation**:
1. During JIT compilation, at each branch target, record vreg types:
   `vreg_type_map[pc][vreg] = { scalar, f32, f64, v128 }`
2. At OSR entry, iterate the map:
   - scalar: load from regs[vreg] to physical reg (existing)
   - v128: load from simd_v128[vreg] to Q register AND regs[vreg]
   - f32/f64: ensure FP cache is correctly populated
3. The OSR prologue becomes type-aware, loading values differently per type

**Estimated effort**: High — touches JIT compiler core, regalloc, and OSR prologue.

### Approach C: Function Splitting (Loop Extraction)

**Idea**: Extract the hot loop body as a separate "inner function" and JIT only
that. The inner function takes the live variables as parameters and returns
results. No OSR needed — it's a regular function call.

**Pros**:
- No state transfer issues — parameters are explicit
- Works with existing JIT infrastructure
- Matches academic "trace compilation" approaches

**Cons**:
- Complex IR transformation: need to identify loop, extract body, generate wrapper
- Live variable analysis at loop boundaries
- v128 parameters need ABI support (already have multi-value return)

**Implementation**:
1. At back-edge threshold, identify the loop body (back_edge_target → back_edge_source)
2. Analyze live variables at loop entry (read before write in loop body)
3. Generate a new RegFunc containing only the loop body + return
4. JIT-compile the extracted function (no guard, no side effects)
5. Call the JIT function from the interpreter, passing live vars as args
6. On return, write results back to interpreter's register file

**Estimated effort**: Very High — new IR transformation pass.

### Approach D: Unified Register File (eliminate simd_v128)

**Idea**: Store v128 values in the main register file (as two consecutive u64
slots) instead of a separate simd_v128 array. This eliminates the sync problem.

**Pros**:
- No sync needed between regs[] and simd_v128[] — single source of truth
- OSR "just works" — all values in one place
- Simpler JIT code for MOV/CONST (no clear/copy needed)

**Cons**:
- Doubles register file size for v128 values (2 slots per vreg)
- Requires changes to interpreter, JIT, trampoline, and regalloc
- v128 load/store needs to handle non-contiguous layout or pair-based addressing
- W37 (contiguous v128 storage) optimization relies on [2]u64 layout in simd_v128

**Implementation**:
1. Change register file from `u64[]` to `u128[]` or use paired slots
2. Update all interpreter ops to read/write paired slots for v128
3. Update JIT to use paired register file slots instead of simd_v128
4. OSR prologue loads from unified register file

**Estimated effort**: Very High — touches everything.

## Recommendation

**Start with Approach A (Lazy AOT)** as the primary path:
- It's the simplest approach that solves the problem completely
- Matches wasmtime's proven architecture
- Already partially designed in Phase 18.2
- Falls back to interpreter for functions too large for JIT

If Approach A proves insufficient (e.g., startup too slow for large modules),
consider **Approach B** (frame state metadata) as a targeted fix for the remaining
cases.
